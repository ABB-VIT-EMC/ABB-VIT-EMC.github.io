saarto Documentation¶

saarto¶

System for Automated, Anti-Redundancy Test Orchestrator or saarto is an automatic test orchestrator that relies on SIEMENS Polarion product testing structure. This program is created in EMC and Smart Building Lab in ABB Vittuone, Italy. Instead of creating the test task one-by-one, the system takes the test case/test logic from from a template file that describe a test structure that is also hardware agnostic, and another set of structural elements (for example a ZERA generator and a M4M 30 Ethernet) to create the final Robot Framework test task. This results in zero-redundancy, easy-to-write, and easy-to-maintainable codebase.

Introduction¶

The Pipeline in a Nutshell (assume that we are about to generate a test task PMD-0000)

Bootstrap (create): The user selects (or the information is obtained via SIEMENS Polarion) a test case (a.k.a. test structure or test logic), e.g., PMD-1111 in a config.toml.

Auto-Resolution (fetch): The system “reads” the selected test template to identify required hardware (e.g., a Power Meter or Generator). It then automatically fetches the corresponding Structural Element templates and their TOML parameters to the afformentioned config.toml (where user (or Polarion) then fill the desired instruments (e.g. PMD-2222, and PMD-3333)).

Build (build): The pipeline merges the test steps, instrument setups, and site-specific parameters into a unified context. This also auto generates instrument.toml for hardware-specific configuration (e.g. for PMD-2222, and PMD-3333) that should be filled before run-time.

Rendering (render): A final Robot Framework file is generated, ensuring that the code is syntactically consistent and perfectly aligned with the target environment. All these files allow us to also generate the PMD-0000 README.md file

For step-by-step workflow:

saarto init PMD-1111 PMD-0000 # linking/bootstrapping the test case PMD-0000 to the new test task PMD-1111
# before fetching, user should fill all the instrument fields in <TASKS_DIR>/PMD-1111/config.toml
saarto fetch PMD-1111 # create <TASKS_DIR>PMD-1111/instrument.toml
saarto build PMD-1111 # compile all the templates resulting in <TASKS_DIR>/PMD-1111/PMD-1111.robot.j2
saarto render PMD-1111 # before running make sure all the data is set, then this will generate the version that parameters are resolved
saarto execute PMD-1111 --skip_build # basically run the generated Robot Framework file and manage the result

In practice, we follow real workflow

saarto init PMD-1111 PMD-0000
# the same as fetching, the user must fill the instrument fields
saarto fetch PMD-1111 # at this point user should edit the configuration instruments in instrument.toml, e.g., port, ip, baudrate, etc.
saarto execute PMD-1111 # withoiut --skip_build this should build, render and then run the generated code
saarto clean PMD-1111 # by running this, we start with a clean slate (deleting PMD-1111.robot), user code will be lost as well.

Installation¶

This project is tested on Python 3.14, for automatic installation, run

pip3 install --upgrade pip setuptools wheel
git pull https://github.com/ABB-VIT-EMC/saarto
pip install -e ./saarto

or use this instead of cloning from GitHub

pip install "saarto @ git+https://github.com/ABB-VIT-EMC/saarto"

For easier installation refer to ATE (Automatic Test Environment)

Dependencies¶

  • AutoTestLib

  • matta

  • pydantic

  • jinja2

  • robotframework The dependencies above should be installed from pyproject.toml except our in-house libraries, which should be install by doing

pip install "matta @ git+https://github.com/ABB-VIT-EMC/matta"
pip install "AutoTestLib @ git+https://github.com/ABB-VIT-EMC/AutoTestLib"

Configuration Resolution Scheme¶

SAARTO uses a two-tier, fallback-based configuration system. This allows individual users or projects to customize settings safely without modifying the core source code, while still guaranteeing that the application works out-of-the-box.

This logic is driven by the get_config_path and load_main_config functions.

How It Works¶

When the application requests a configuration file (e.g., saarto.toml), the system attempts to locate it in the following order:

1. User/Virtual Environment Config (Priority)¶

First, the system checks if you are running inside a Python Virtual Environment (sys.prefix). It looks for a config/ folder located right next to your active virtual environment directory:

  • Path: <venv_parent_dir>/config/<config_file>

  • Why? This allows developers to easily inject custom parameters, paths, or logging levels specific to their current workspace, without overriding the global application defaults.

2. System/Default Config (Fallback)¶

If no user-specific configuration is found next to the virtual environment, the system automatically falls back to the default configuration bundled directly inside the SAARTO package source code:

  • Path: <saarto_source_dir>/core/config/<config_file>

  • Why? This serves as the “safe default.” It guarantees that the pipeline always has a valid baseline configuration to run against, even on a fresh installation.

Summary¶

By using load_main_config("saarto.toml"), the API gracefully handles environment-specific overrides while maintaining a solid default state. If neither file exists, it raises a clear FileNotFoundError to halt execution before any unexpected behavior occurs.

Structural Element Import¶

There is also a possibility to import structural elements from another library like AutoTestLib.

Module’s side code

### TOML-BLOCK-START: INSTRUMENT_DEFINITION
[AutoTestLib__m4m_tcp]
class = "abb_power_meter"
.
.
### TOML-BLOCK-END: INSTRUMENT_DEFINITION

### ROBOT-BLOCK-START: SE_DEPENDENCIES
Library    AutoTestLib.Interfaces.M4MModbusTCP    AS    {{ AutoTestLib__m4m_tcp.name }}
### ROBOT-BLOCK-END: SE_DEPENDENCIES

saarto’s side (instrument fields in config.cmconf, that was fetched from testcase)

[instrument]
generic_power_meter = "AutoTestLib__m4m_tcp"

with “__” double underscore, saarto knows that the word at the front is a python module with saarto templates inside and should be imported.

SAARTO Core API: Pipeline & Query¶

The SAARTO Python API is designed around a clear separation of concerns. All interactions with the framework are routed through two main classes: api.Pipeline and api.Query.

Think of Pipeline as the “actor” that creates and runs things, and Query as the “observer” that reads and searches for things.

1. api.Pipeline¶

The Pipeline class is responsible for the entire lifecycle of a test task. It handles everything that involves creating, mutating, or executing files on the disk.

Every method in the Pipeline returns a Result object (or a list of them), which provides a standardized way to check for success, warnings, or errors.

Key Responsibilities:

  • init / load_config: Creates new test task directories and injects configurations from API payloads or templates.

  • fetch: Pulls the required hardware/instrument definitions from Structural Elements.

  • build & render: Compiles the Robot Framework (.robot) files using Jinja2 templates and dynamic data.

  • execute: Triggers the Robot Framework runner and packages the results securely into hashed .tar.gz archives.

  • clean: Removes temporary build files.

Quick Example:

from saarto import api

# Complete a test lifecycle
api.Pipeline.init(test_name="PMD-1234", test_case="PMD-0000")
api.Pipeline.fetch("PMD-1234")
api.Pipeline.build("PMD-1234")

# Execute and check the result
result = api.Pipeline.execute("PMD-1234", test_instance="nightly_run")
if result[0].ok():
    print("Execution successful!")

Test Result Management¶

When a test task is executed, the SAARTO pipeline manages the outputs and artifacts to ensure that every test run is fully traceable, immutable, and reproducible. This is handled through a structured directory naming convention and cryptographic archiving.

Execution and Directory Structure¶

When you trigger test execution via the pipeline, it automatically organizes the output files. A unique, time-stamped parent directory is created for the execution instance.

For example, if you execute a batch of tests with the instance name "release_candidate", the pipeline creates an isolated directory like: 2026-05-06_14-30-00_release_candidate/

Inside this directory, individual test tasks output their Robot Framework logs, reports, and a hashed archive of their configurations.

How Results are Archived¶

At the end of a test execution, the pipeline preserves the exact state of the test inputs using a cryptographic hashing mechanism.

  1. Gather Artifacts: The pipeline collects all relevant source files used for that specific execution (e.g., config.toml, instrument.toml, temp.toml).

  2. Compression: These files are compressed into a single .tar.gz archive.

  3. Cryptographic Hashing: A SHA-256 hash is generated based on the exact byte-content of the archive.

  4. Unique Identification: The archive is renamed using the first 8 characters of the SHA-256 hash (e.g., a1b2c3d4.tar.gz) and saved to the specific output directory for that test instance.

Siemens Polarion Integration¶

This archiving step is critical for syncing test results back to our Application Lifecycle Management tool, Siemens Polarion.

When the pipeline patches and attaches the test result to Polarion, it reads the configuration data directly from this hashed .tar.gz archive rather than the active local workspace.

Why do we do this?

  • Resistance to Change: If an engineer edits the local test configuration immediately after a test finishes but before the results are synced, the uploaded result remains completely unaffected.

  • Decoupled from Local State: The upload process does not depend on the live, local test folders. It relies entirely on the frozen snapshot captured at the exact moment of execution.

  • Traceability: The 8-character hash acts as a unique fingerprint. If someone needs to debug a test run months later, the hash guarantees exactly which configurations and instruments were used.

Error Handling, Custom Dataclasses, and Exceptions¶

In the SAARTO pipeline, we use two main tools to handle outcomes and errors: the Result dataclass and custom Exceptions.

1. The Result Dataclass¶

Instead of returning True/False, pipeline functions return a Result object. This tells the CLI and GUI exactly what happened so they can show the right messages and colors.

Key fields:

  • code (int): 0 means success. Any other number means failure or warning.

  • status (str): A short label like "SUCCESS", "ERROR", or "WARNING".

  • message (str): The text to display to the user.

How to check it: Because Result has a special boolean method, you can just use if result: to check if it was successful.

2. Custom Exceptions¶

Sometimes things go critically wrong (e.g., a required file is missing or contains invalid syntax). Instead of returning a Result, we raise a custom exception.

  • PipelineError: The base error for everything.

  • TestCaseNotFoundError: Raised when a test case file doesn’t exist.

  • ConfigurationError: Raised when a config file is missing or has broken TOML syntax.

The CLI and GUI catch these exceptions so the app doesn’t crash, and instead shows a clean error banner to the user.

Simple Examples¶

Here is a simple example showing when to return a Result versus when to raise an exception:

from saarto.core.dataclass import Result, TestCaseNotFoundError

def run_test(test_name: str) -> Result:
    # 1. CRITICAL ERROR: Raise an Exception
    if not test_name:
        raise TestCaseNotFoundError("No test name provided! Cannot proceed.")

    # 2. EXPECTED FAILURE: Return a Result
    if test_name == "broken_test":
        return Result(code=1, status="WARNING", message="Test ran, but had warnings.")

    # 3. SUCCESS: Return a Result
    return Result(code=0, status="SUCCESS", message="Test finished perfectly.")

from pathlib import Path
import tomllib
from saarto.core.dataclass import ConfigurationError

def load_config(path: Path):
    if not path.exists():
        # Raise an exception because the program cannot proceed without config
        raise ConfigurationError(f"Config file not found: {path}")
        
    try:
        with open(path, "rb") as f:
            return tomllib.load(f)
    except tomllib.TOMLDecodeError as e:
        raise ConfigurationError(f"Invalid TOML syntax: {e}")
